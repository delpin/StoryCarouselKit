# Архитектура и мотивация

## Обзор архитектуры

Story Carousel построен как монорепозиторий с четким разделением ответственности между пакетами. Архитектура следует принципам **framework-agnostic** разработки, где бизнес-логика отделена от UI-компонентов.

```
story-carousel-monorepo/
├── packages/
│   ├── native/           # 🔧 Ядро бизнес-логики
│   ├── react/            # ⚛️  React обертка
│   ├── vue/              # 🟢 Vue обертка (планируется)
│   ├── svelte/           # 🧡 Svelte обертка (планируется)
│   └── angular/          # 🅰️  Angular обертка (планируется)
└── docs/                 # 📚 Документация
```

## Мотивация

### Почему монорепозиторий?

1. **Единое управление зависимостями** — общие dev-зависимости и скрипты сборки
2. **Совместное использование кода** — нативное ядро используется всеми фреймворками
3. **Синхронизированные релизы** — все пакеты обновляются одновременно
4. **Общая документация и тесты** — единые стандарты качества

### Почему framework-agnostic?

1. **Максимальная гибкость** — один компонент для всех проектов
2. **Легкая миграция** — смена фреймворка без переписывания логики
3. **Изоляция логики** — UI и бизнес-логика не связаны
4. **Упрощенное тестирование** — тестируется только логика, не UI

## Архитектурные принципы

### 1. Single Source of Truth

Вся бизнес-логика сосредоточена в `@storycarouselkit/core`. Обертки фреймворков только адаптируют UI.

### 2. Immutable State

Состояние компонента иммутабельно. Все изменения происходят через явные методы API.

### 3. Event-Driven Architecture

Компонент использует событийную модель для коммуникации с внешним миром через коллбэки.

### 4. Type Safety

Полная TypeScript поддержка во всех пакетах с shared типами.

## Пакеты и их роли

###@storycarouselkit/core

**Ответственность:**

- Управление состоянием историй
- Логика таймера и прогресса
- Навигация между историями
- Событийная система

**Не отвечает за:**

- Рендеринг UI
- Стилизацию
- Интеграцию с фреймворками

### Framework-обертки (@storycarouselkit/react, etc.)

**Ответственность:**

- Адаптация нативного API под конкретный фреймворк
- Рендеринг UI компонентов
- Управление жизненным циклом
- Интеграция с экосистемой фреймворка

**Не отвечают за:**

- Бизнес-логику
- Управление состоянием
- Событийную систему

## Data Flow

```
User Interaction → Framework Wrapper → Native Core → State Update → UI Re-render
       ↑                                                            ↓
   Events/Callbacks ← Framework Wrapper ← Native Core ← State Change
```

## Преимущества подхода

### Для разработчиков

- **Простота использования** — знакомый API для каждого фреймворка
- **Предсказуемость** — единая логика поведения
- **Надежность** — хорошо протестированное ядро

### Для поддерживаемости

- **Модульность** — изменения в одном пакете не ломают другие
- **Тестируемость** — бизнес-логика тестируется независимо
- **Расширяемость** — легко добавить новый фреймворк

### Для производительности

- **Минимальный бандл** — только нужный фреймворк
- **Оптимизированные перерендеры** — контролируемые обновления UI
- **Эффективное управление памятью** — явная очистка ресурсов

## Будущие расширения

Архитектура позволяет легко добавлять новые возможности:

- **Новые фреймворки** — Solid, Qwik, Lit
- **Новые типы контента** — видео, интерактивные истории
- **Расширенная аналитика** — метрики просмотров, вовлеченности
- **Оффлайн поддержка** — кэширование историй
